  - Auth: None
  - Body: { email, password, role? }
  - Validation: email valid, password required
  - Note: If role is provided, verifies user's role matches (prevents
          employee logging into employer portal and vice versa)
  - Response: { token, user: { ... } }

POST /api/auth/google
  - Purpose: Google OAuth login/registration
  - Auth: None
  - Body: { credential (Google ID token), role?, company? }
  - Flow:
    1. Verifies Google token with Google OAuth2Client
    2. Extracts email, name, picture, googleId
    3. If user exists: validates role, returns token
    4. If new user without role: returns { newUser: true, email, name }
       (frontend shows role picker)
    5. If new user with role: creates user, returns token
  - Response: { token, user: { ... } } OR { newUser: true, email, name }

POST /api/auth/logout
  - Purpose: Clear auth cookie
  - Auth: None
  - Response: { message: "Logged out successfully" }

GET /api/auth/me

3. GEMINI AI INTEGRATION
================================================================================

File: server/utils/resumeParser.js

Setup:
------
  const { GoogleGenerativeAI } = require("@google/generative-ai");
  const genAI = new GoogleGenerativeAI(process.env.GEMINI_API_KEY);
  const model = genAI.getGenerativeModel({ model: "gemini-2.5-flash" });

Model Used: gemini-2.5-flash (fast, optimized for rapid processing)

How Gemini is Used - PDF Text Extraction:
-----------------------------------------
Function: parsePDFGemini(buffer, retries = 2)

1. PDF file buffer is converted to base64
2. Sent to Gemini API as inline data with mimeType "application/pdf"
3. Prompt sent to Gemini:

   "Extract ALL text content from this resume PDF exactly as it appears.
    Include every section, every detail - name, contact info, education,
    experience, skills, projects, certifications, everything.
    Return ONLY the extracted text, no commentary or formatting changes."

4. Gemini returns plain text extraction of resume content
5. If rate limited (429 error), retries with delays (10s, 20s)

Fallback Strategy (if Gemini fails):
-------------------------------------
1. Try pdf-parse v1 (works on Vercel serverless - no native bindings)
2. Try pdf-parse v2 (requires native bindings, better quality locally)
3. Return null if all fail

Integration with Resume Upload:
-------------------------------
1. PDF uploaded via multer
2. Buffer sent to Gemini for text extraction
3. Extracted text stored in database (parsedText field)
4. Text immediately analyzed for ATS scoring
5. On Vercel: Buffer used directly from memory
6. Locally: File saved to disk, buffer read from file

---------------------

  6. DELETION:   File removed from disk (local), database record deleted

6. AUTHENTICATION & SIGNUP/LOGIN SYSTEM
================================================================================

6.1 JWT Token System
---------------------
File: server/middleware/auth.js

- Token Generation: jwt.sign({ id }, JWT_SECRET, { expiresIn: "70d" })
- Token stored in:
  a) httpOnly cookie (secure in production, sameSite: lax)
  b) JSON response body (stored in localStorage by frontend)
- On each request, token checked from:
  a) Authorization: Bearer <token> header
  b) Cookie

6.2 Registration Flow
----------------------
1. User fills form: name, email, password, role (employee/employer)
2. POST /api/auth/register
3. Backend validates inputs (express-validator)
4. Checks if email already exists
5. Password hashed with bcryptjs (12 salt rounds)
6. User document created in MongoDB
7. JWT token generated
8. Token set as cookie + returned in response
9. Frontend stores token in localStorage
10. User redirected to dashboard

6.3 Login Flow (Email/Password)
-------------------------------
1. User enters email + password + selects role tab
2. POST /api/auth/login with { email, password, role }
3. Backend finds user by email
4. Password verified with bcrypt.compare()
5. If role provided, verifies user's role matches login portal
   (prevents employee logging into employer portal)
6. JWT token generated and returned
7. Frontend stores token, redirects to appropriate dashboard

6.4 Google OAuth Flow
---------------------
1. User clicks "Sign in with Google" button
2. Google OAuth popup appears (via @react-oauth/google)
3. Google returns ID token (credential)
4. POST /api/auth/google with { credential, role?, company? }
5. Backend verifies token with Google's OAuth2Client
6. Extracts: email, name, picture, googleId (sub)
7. Three scenarios:
   a) User exists by googleId -> Login, return token
   b) User exists by email -> Link googleId, login
   c) New user WITHOUT role -> Return { newUser: true, email, name }
      Frontend shows role selection dialog
   d) New user WITH role -> Create account, return token

6.5 Session Persistence
------------------------
- On app load, AuthContext calls GET /api/auth/me
- Token sent from localStorage via Axios interceptor
- If valid: user data loaded, app shows authenticated state
- If invalid/expired: token cleared, redirect to login

6.6 Logout
-----------
- POST /api/auth/logout clears cookie
- Frontend removes token from localStorage
- User state set to null
- Redirect to home page

6.7 Role-Based Access Control
------------------------------
File: server/middleware/role.js

- requireRole(...roles) middleware function
- Checks req.user.role against allowed roles
- Returns 403 "Access denied" if unauthorized
- Used on employer routes: requireRole("employer")

6.8 Password Security
----------------------
- Hashed with bcryptjs, 12 salt rounds
- Password field has select: false (excluded from queries by default)
- Pre-save hook: only re-hashes if password was modified
- Password change requires old password verification
- Minimum 6 characters

8. MIDDLEWARE
================================================================================

8.1 auth.js - JWT Authentication Middleware
-------------------------------------------
- Checks Authorization header (Bearer token) OR cookie
- Verifies JWT signature with JWT_SECRET
- Loads user from database by decoded ID
- Sets req.user for downstream route handlers
- Returns 401 if no token or invalid

8.2 role.js - Role-Based Access Middleware
-------------------------------------------
- Factory: requireRole(...roles)
- Checks req.user.role against allowed roles array
- Returns 403 "Access denied" if role not in allowed list

8.3 upload.js - File Upload Middleware (Multer)
-----------------------------------------------
Two upload handlers:

uploadAvatar:
  - Field name: "avatar"
  - Max size: 2MB
  - Allowed: JPEG, JPG, PNG, WEBP
  - Storage: disk (local) or memory (Vercel)

uploadResume:
  - Field name: "resume"
  - Max size: 5MB
  - Allowed: PDF, DOC, DOCX
  - Storage: disk (local) or memory (Vercel)

Vercel detection: process.env.VERCEL === "1"
